## Executive Summary

| File | Purpose | Tables | Primary ID Type | Approach |
|------|---------|--------|----------------|----------|
| `supabase_migration.sql` (v2.0) | Base production schema | 16 + 5 views | `BIGINT` (identity) + `UUID` for auth | Idempotent, IF NOT EXISTS, 3NF |
| `supabase_migration_v3.sql` | Additive upgrade (run AFTER v2.0) | 7 new tables + ALTERs | Same as v2.0 | ALTER TABLE ADD COLUMN IF NOT EXISTS |
| `migration.sql` | Standalone alternative schema | 17 tables | `UUID` (gen_random_uuid) everywhere | DROP-CREATE, non-idempotent |
| `migration_runner.cjs` | Node.js execution of migration.sql DDL | 19 tables | `UUID` | Programmatic DROP + CREATE |

### ⚠️ CRITICAL CONFLICT ALERT
There are **SEVERE schema incompatibilities** between the v2.0/v3.0 Supabase migration path and the `migration.sql` / `migration_runner.cjs` path. They use **different primary key types** (`BIGINT` vs `UUID`), **different column names**, and **different foreign key relationships**. You **cannot mix** these two schemas on the same database.

## Part 5: Conflict & Compatibility Analysis

### 5.1 The Two Competing Schema Families

| Aspect | v2.0 + v3.0 Path | migration.sql / runner.cjs Path |
|--------|-----------------|--------------------------------|
| **Primary Keys** | `BIGINT` (identity) for business tables, `UUID` only for auth/profiles | `UUID` (gen_random_uuid) for ALL tables |
| **FK References** | `BIGINT` → `BIGINT` (e.g., members.id ↔ trainers.id) | `UUID` → `UUID` |
| **Auth Linking** | `profiles.id` → `auth.users(id)` | Direct `auth_user_id` in members/trainers |
| **Audit Columns** | `created_by` / `updated_by` → `profiles(id)` | Sometimes `logged_by` / `executed_by` / `actor_id` → `auth.users(id)` |
| **Settings** | `key` TEXT PK, `value` TEXT | `key` VARCHAR PK, `value` JSONB |
| **Deleted At** | Most tables have `deleted_at` soft delete | Some tables missing `deleted_at` |
| **Computed Columns** | `days_remaining`, `status` in `members_view` | Stored `status` column in `members` |
| **Idempotency** | `IF NOT EXISTS`, `ON CONFLICT` — safe to re-run | `DROP TABLE CASCADE` — **DESTROYS DATA** |
| **Views** | 4 views (`members_view`, `payments_view`, `attendance_view`, `reminder_logs_view`) | **NO views** |
| **Check Constraints** | Extensive CHECK constraints on enums | Minimal CHECK constraints |
| **Indexes** | Partial indexes (WHERE deleted_at IS NULL) | Simple indexes only |

## Part 7: Recommendations

### 7.1 Choose ONE Schema Path

You **must** decide between:

**Option A: v2.0 + v3.0 (Recommended)**
- Richer feature set (membership_plans, profiles, views, measurements, activity_logs)
- Better normalization (3NF enforced via views)
- Idempotent (safe to run multiple times)
- BIGINT IDs are more compact and human-readable
- Better CHECK constraints and partial indexes
- Soft delete pattern on most tables
- Comprehensive audit trail (`created_by`, `updated_by`)

**Option B: migration.sql / runner.cjs**
- UUIDs everywhere (good for distributed systems, but overkill for single-gym app)
- Simpler structure but less feature-complete
- DROP-CREATE is destructive
- Missing many business tables (profiles, membership_plans, measurements)
- Less constraint enforcement

### 7.2 If You Choose v2.0 + v3.0

1. **Delete or archive** `migration.sql`, `migration_runner.js`, `migration_runner.cjs` to prevent accidental execution
2. **Remove** hardcoded credentials from all files
3. **Implement** the production RLS checklist (Section 17 comments)
4. **Move** `api_key` from `whatsapp_config` to Supabase Vault
5. **Add** `REPLICA IDENTITY FULL` to tables that need realtime (currently only using `ALTER PUBLICATION`, may need REPLICA IDENTITY for UPDATE operations)
6. **Back up** before running v3.0 ALTER statements on a populated database

### 7.3 If You Choose migration.sql / runner.cjs

1. **Add** `profiles` and `membership_plans` tables to match app expectations
2. **Add** `members_view` or change app to compute `days_remaining` client-side
3. **Add** soft delete columns (`deleted_at`) to all mutable tables
4. **Add** `created_by` / `updated_by` audit columns
5. **Change** id generation from `gen_random_uuid()` to `bigserial` if you want to match the v2.0 app design
6. **Add** CHECK constraints for data quality

### 7.4 Immediate Actions

1. **Rotate the Supabase password** — it's exposed in Git history
2. **Remove** `migration_runner.js` and `migration_runner.cjs` from the repo or sanitize them
3. **Verify** which schema is actually live in Supabase by running `server/verify_schema.js` with proper `.env` credentials
4. **Check** `server/src/db.js` and `server/src/supabase.js` to see which column names they expect (this will reveal which schema the backend is written for)
